# Property Structure Documentation

## Overview
The project now uses a two-tier property system combining SAG DevOps CI Assets reference properties with custom project-specific properties.

**Status**: Ō£ģ IMPLEMENTED

## ­¤ōü Property Files

### 1. build.properties (Reference/Standard)
**Source**: SAG DevOps CI Assets master_build_Reference  
**Purpose**: Standard SAG build properties  
**Location**: `build.properties`  
**Modification**: Should NOT be modified (reference file)

**Key Properties**:
```properties
sag.install.dir=${SAGHome}
build.output.dir=${buildOutputDir}
build.source.dir=${isProjectsDir};${isConfigDir};...
enable.build.IS=true
enable.build.MWS=false
build.log.enable=true
```

---

### 2. custom.properties (Project-Specific)
**Purpose**: Project-specific overrides and additions  
**Location**: `custom.properties`  
**Modification**: Edit this file for project customizations

**Key Properties**:
```properties
# Project Info
bda.projectName=hsi_onprem_cicd
bda.buildNumber=local

# Paths
SAGHome=C:/IBM/webMethods
isPackagesDir=packages
isTestDir=tests

# Deployer
config.deployer.deployerHost=localhost
config.deployer.deployerPort=5555
config.deployer.deployerUsername=Administrator
config.deployer.deployerPassword=manage

# Variable Substitution
config.deployer.doVarSub=true
varsubDir=varsub
config.deployer.splitDeploymentSets=false
```

---

### 3. Workflow-Generated Properties (CI/CD)
**Purpose**: Runtime properties generated by GitHub Actions  
**Location**: Generated dynamically (e.g., `deploy.properties`)  
**Modification**: Generated automatically, not edited manually

**Example**:
```properties
SAGHome=$env:SAG_HOME
bda.targetEnv=DEV
bda.buildNumber=${{ github.run_number }}
config.deployer.deployerHost=${{ secrets.DEPLOYER_HOST }}
```

## ­¤öä Property Loading Order

Properties are loaded in this sequence (later overrides earlier):

```
1. build.properties (Standard SAG properties)
   Ōåō
2. custom.properties (Project-specific overrides)
   Ōåō
3. Workflow properties (CI/CD runtime, if specified)
   Ōåō
4. Command-line -D parameters (Highest priority)
```

### Implementation in build.xml:
```xml
<!-- 1. Load standard SAG DevOps build.properties -->
<property file="${basedir}/build.properties"/>

<!-- 2. Load custom project-specific properties -->
<property file="${basedir}/custom.properties"/>

<!-- 3. Load workflow-generated properties if specified -->
<property name="project.properties" value=""/>
<property file="${basedir}/${project.properties}"/>
```

## ­¤Ä» Usage Scenarios

### Scenario 1: Local Development
```bash
# Uses build.properties + custom.properties
cd cicd
ant -f build.xml build
```

**Properties Used**:
1. build.properties (standard)
2. custom.properties (localhost defaults)

---

### Scenario 2: Local with Overrides
```bash
# Override specific properties
ant -f build.xml deploy \
  -DSAGHome=/opt/softwareag \
  -Dbda.targetEnv=DEV
```

**Properties Used**:
1. build.properties (standard)
2. custom.properties (project-specific)
3. Command-line overrides (highest priority)

---

### Scenario 3: CI/CD Deployment
```yaml
# GitHub workflow generates deploy.properties
ant -f build.xml deploy -Dproject.properties=deploy.properties
```

**Properties Used**:
1. build.properties (standard)
2. custom.properties (project-specific)
3. deploy.properties (workflow-generated, uses secrets)

## ­¤ö¦ Customization Guide

### To Add New Property:

**Option A: Project-Wide (Recommended)**
Add to `custom.properties`:
```properties
my.custom.property=value
```

**Option B: Environment-Specific**
Add to `deployer/environments/ENV.properties`:
```properties
MY_ENV_PROPERTY=value
```

**Option C: Workflow-Specific**
Add to workflow property generation:
```yaml
my.workflow.property=value
```

## ­¤öŹ Property Resolution Example

**Example**: Resolving `config.deployer.deployerHost`

```
1. build.properties: (not defined)
2. custom.properties: config.deployer.deployerHost=localhost
3. deploy.properties: config.deployer.deployerHost=${{ secrets.DEPLOYER_HOST }}

```
