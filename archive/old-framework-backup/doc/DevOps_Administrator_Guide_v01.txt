2.0	INTRODUCTION
2.1	Purpose of this document

Fruit Egypt is using IBM's webMethods platform for mission critical enterprise integrations for 
both internal and external data exchanges. This landscape has grown over a period with more 
interfaces and system adapters were being added to the platform regularly, making it 
challenging to manage the delivery of integration projects. With changing business and I.T 
landscape, is looking for an effective continuous delivery and automation setup to reduce time to 
on-board integration solutions and improve quality of deliverables through automated testing 
and deployment. 

They key objectives of this setup are,

?	To reduce delivery timelines and improve quality of deliverables by adopting automated 
quality checks and end-to-end deployment process.

?	To reduce friction between DEV, Ops and QA teams for deployments and release 
management thereby creating a DevOps delivery approach.

?	To establish an automated, reliable, repeatable integration projects delivery mechanism that 
supports agile delivery of integration projects.

?	To minimize the deployment errors generally caused with human intervention during he 
process of deploying the assets.

?	To provide an easy rollback mechanism.
2.2	Audience

*	WebMethods Developers.
*	Development Managers.
*	Technical architect

3.0	DEVOPS
3.1	Overview 

The need for DevOps arose from the increasing success of agile software development, as that 
led to organizations wanting to release their software faster and more frequently. As they sought 
to overcome the strain this put on their release management processes, they had to adopt 
patterns such as application release automation, continuous integration tools, and continuous 
delivery.

      


 
3.2	Continuous Integration (CI)
They goal of CI is to provide rapid feedback so that if a defect is introduced into the code 
base, it can be identified and corrected as soon as possible. CI means that one developer 
who writes code on his laptop at home, and another dev who codes on her desktop in the office 
can both write software for the same product separately, integrate their changes together in a 
place called the source repository. They can then build the combined software from the bits they 
each wrote and test that it works the way they expect.
CI requires that all developers have self-testing code. This is code that tests itself to ensure that 
it is working as expected, and these tests are often called unit tests. When all the unit tests pass 
after the code is integrated and will get a green build. This indicates that they have verified that 
their changes are successfully integrated together, and the code is working as expected by the 
tests. However, while the integrated code is successfully working together, it is not yet ready for 
production because it has not been tested and verified as working in production-like 
environments. To be practicing CI, developers must check-in to the main source repository, 
integrate and test their code frequently and often. Usually many times an hour, but at a 
minimum once a day.

 
         

3.3	Continuous Delivery (CD)
Continuous Delivery means that each time developers changes to the code, integrates and 
builds the code, that they also automatically test this code on environments that are very similar 
to production. We call this progression of deploying to - and testing on - different environments 
a deployment pipeline. Often the deployment pipeline has a development environment, a test 
environment, and a staging environment, but these stages vary by the team, product and 
organization.


   	        

3.4	Continuous Deployment
This is a practice where every change that developers make, and which passes all the test 
stages, automatically goes to production. Some companies do this, and others do not. To 
achieve continuous deployment, you first need to get to continuous delivery, so it's not a priority 
to decide which is better for you until you've starting practicing CD

		

3.5	Logical View - how does it work
It is important to have a build server for practicing CI. It is equally important that either unit tests 
or integration tests are treated as codebase and checked in to the VCS so that they can be 
tested during CI process. Also, CI Process should have the dash boarding capability to show the 
code review and test reports.

             

       Below are the high-level steps in the continuous integration process.

1.	Checkin the assets from the developer's local designer to VCS (Git Lab)

2.	Developers must daily update their local repository from VCS (Git Lab) before making changes to the 
code. This will ensure everyone is working on the latest copy of the code

3.	Jenkins Job will checkout the code on the scheduled manner

4.	Using IBM's asset build environment and deployer scripts, Jenkins will create the build and deploys to 
the Target server.

5.	Unit test cases are executed on the deployed assets and reports will be shared with relevant stake 
holders

6.	If the build is stable with the desired quality, it will be promoted to the higher environments 
(Test/PreProd/PROD)

This Jenkins job will do the checkout from non-prod GITLab branch of a particular business 
application i.e. DEV_{BusinessApplication}, example DEV_ADSL_NONL branch.  

        
       
Post checkout, it will trigger another jenkins job to identify the changes i.e. 
{BusinessApplication}_identify_changes , example ADSL_NONL_identify_changes.

 


4.5.2	{BusinessApplication}_identify_changes
Once the checkout from GITLab is done by job {BusinessApplication}/{BusinessApplication_ 
checkout_git  e.g  ADSL/ADSL checkout_git, this job {BusinessApplication}_identify_changes e.g. 
ADSL_NONL_identify_changes will be executed. This unix scripted Jenkins job will identify the 
packages which are modified as part of last commit made in GITLab branch and will place the 
modified packages into a different workspace directory uniquely identified by commitID. Below is the 
script call and parameters passed to it.  

/wMDevops/apps/cicd/scripts/getISChanges.sh ${checkoutPath} ${lastGITCommitID} 
${BusinessApplication}/wMDevops/apps/cicd/scripts/getISChanges.sh ${checkoutPath} 
${lastGITCommitID} ${BusinessApplication}

        
       
As part of post processing action, it will trigger another jenkins job to create the build of changed 
packages i.e. {BusinessApplication}_build_abe, example ADSL_NONL_build_abe.

 


4.5.3	{BusinessApplication}_build_abe
This Jenkins job {BusinessApplication}/{BusinessApplication}_ build_abe  e.g  
ADSL/ADSL_build_abe will create the build.properties file at run time  based on the lastGITCommit. 
It will call script below:
==================================================================
echo "lastGITCommitID: $lastGITCommitID"

cp ${abeBuildDir}/REFERENCE_build.properties ${abeBuildDir}/build.properties

deployerProjectName=`echo ${lastGITCommitID} | cut -c 1-10`_IS_${BusinessApplication}
lastGItCommitIDWith10Char=`echo ${lastGITCommitID} | cut -c 1-10`

echo "#Added parameters at run time for GIT Commit ID: ${lastGITCommitID}" >> ${abeBuildDir}/build.properties
echo "build.output.dir=${WORKSPACE}/output/${lastGItCommitIDWith10Char}" >> ${abeBuildDir}/build.properties
echo "build.source.dir=${BusinessApplicationWorkspace}/ISChanges/${lastGItCommitIDWith10Char}" >> 
${abeBuildDir}/build.properties
echo "build.archive.dir=${WORKSPACE}/archive/${lastGItCommitIDWith10Char}" >> ${abeBuildDir}/build.properties
echo "build.log.fileName=${WORKSPACE}/logs/${lastGItCommitIDWith10Char}/build.log" >> ${abeBuildDir}/build.properties

4.8	How to add new Business Application in Automation 

4.8.1	Add Jenkins Jobs
Follow below steps for adding a new business application in Jenkins

>	Go to Jenkins job dir /wMDevops/apps/jenkins/jobs.
cd /wMDevops/apps/jenkins/jobs
 

>	Create a copy of Business application folder for target, e.g. from ADSL_NONL to ADSL_ONL
cp -pR ADSL_NONL ADSL_ONL
 

>	Go inside the target directory e.g. ADSL_ONL
cd ADSL_ONL
 

>	Run the command below to replace the values of ADSL_NONL to target ADSL_ONL
sed -i 's/ADSL_NONL/ADSL_ONL/g' config.xml
 

>	Verify with grep command if the target name is replaced.
grep ADSL_ONL config.xml
 

>	Go inside jobs dir.
cd jobs
 

>	Run the commands below (make sure to modify the paramters)
for JENKINS_JOB in `ls`;   
do  
sudo rm -rf ./${JENKINS_JOB}/builds;   
sudo rm -rf ./${JENKINS_JOB}/nextBuildNumber;
 sudo sed -i 's/ADSL_NONL/ADSL_ONL/g' ./${JENKINS_JOB}/config.xml; 
 NEW_JENKINS_JOB=`echo ${JENKINS_JOB} |sudo  sed -e 's/ADSL_NONL/ADSL_ONL/g'`;
 sudo mv  ./${JENKINS_JOB} ./${NEW_JENKINS_JOB};  
 done
 

>	Run the command below to verify that on occurrences of old business application exists.
find . -type f |xargs grep -i ADSL_NONL
 

>	Verify that folder names are also changed with target application name e.g. ADSL_ONL
 

>	Restart the Jenkins server
sudo systemctl restart Jenkins
 

>	Verify business application folder from Jenkins UI
 

>	Verify Jenkins jobs inside target folder.
 

>	Done
