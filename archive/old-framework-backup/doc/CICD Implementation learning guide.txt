In this exercise you will learn committing(check-in) package in local IS and to Git
Start the Designer using Run as administrator.
If you do not have admin privilege, you need to do few more steps to commit IS package to Git 
Clone Git Repository
Open Git perspective in Designer
 
Clone your Git repo (Bitbucket or any git repository)
 
 
Create new package in local IS and commit to Git
Start local IS and connect using Designer. Below highlighted icon shows you are using local service development
 
If you do not have admin privilege disable below in preferences
 
Create a package and right click to create local service development project
 

Select the cloned repository in the next window
 
Open java perspective and you will see the package created as a java project
 
Right click the project Team > Add to Index

Exercise 2: Build IS assets using ABE

In this exercise you will learn how to build IS assets using ABE.
The Asset Build Environment (ABE) supplies the scripts necessary to build assets into composites 
that Deployer then deploys to target servers and groups.
Important build.properties
	
You can locate the properties files for the Asset Build Environment in the following directory:
SoftwareAG_directory \common\AssetBuildEnvironment\master_build

*	build.properties Contains the settings that the build script uses to build the assets. You must set the 
properties for the build before running the build script

Property
Description
sag.install.dir
The path to the installation directory of the Software AG products
build.source.dir
A list with the full paths to the source directories that contain 
assets to build. Use a forward slash "/" as path separator. All 
source directories in the list must contain project directories (such 
as IS packages, CAF projects, and TN exports) as direct children. 
Use ";" as the delimiter when listing more than one path. 
For example:
build.source.dir=/<root_path>/<project_name>/IS;/<root_pat
h>/<project_name>/TN

build.output.dir
The root directory in which the build script will place the output 
(composites and descriptors) of the build. Use a forward slash "/" 
as path separator.
For example:
build.output.dir =<root_path>/build/

build.source.project.dir
A list of paths to the project directories that contain the assets to 
build. Use a forward slash "/" as path separator. This property is 
different from build.source.dir, because you use it to define 
individual project directories. Use ";" as the delimiter when listing 
more than one path
enable.build.ProductID
With this property, you can enable or disable building the 
composites for specific runtimes included in the build. Valid values 
are true (build composites for the specified runtime) 
and false (exclude the specified runtime from the build). Each 
runtime has a separate property that you must specify on a 
separate line
For example:
enable.build.UniversalMessaging=true
enable.build.IS=true
Executing build script

The build script can be located in below directory:
SoftwareAG_directory \common\AssetBuildEnvironment\bin
*	build.bat or build.sh The build script files for Windows and Unix systems.
Just executing the script build.bat will take the properties from 
SoftwareAG_directory\common\AssetBuildEnvironment\master_build\build.properties
You can also execute the script by passing the properties as arguments as shown below
build.bat -D<propertyKey>=<propertyValue>
Command syntax:
<SAG_Install_Directory>/common/AssetBuildEnvironment/bin/build.bat
-Dbuild.output.dir=<Build_Output_Directory>
-Dbuild.source.project.dir=<Build_Source_Packages_Semicolon_Separated> (or)
-Dbuild.source.dir=<Build_Source_Directory> 
-Denable.build.IS=true
Example command:
C:/Programs/SoftwareAG/common/AssetBuildEnvironment/bin/build.bat -
Dbuild.output.dir=C:\Users\WDAGUtilityAccount\workspaces\cicd-workspace\builds -
Dbuild.source.project.dir=C:\Users\WDAGUtilityAccount\workspaces\cicd-
workspace\deployerSource\SOALearnings -Denable.build.IS=true

Ant script to update ProjectAutomator.xml

The ProjectAutomator.xml created will be used for deployment of different assets to different target servers.
The content will be dynamically updated using ant script.
The ant script is located in below directory:
<SAG_Install_Directory>/common/lib/ant/bin

*	ant.bat or ant.sh The ant script files for Windows and Unix systems.
The xml config to be passed to the ant script is already created inside the Test Suite Executor project
/CICDDemo/deployer/updateAutomator.xml
Below screenshot shows the sample updateAutomator.xml with comments describing the significance of the 
fields.
 
Ant script uses the above xml to identify the fields which must be replaced in Deployment_ProjectAutmator.xml 
For Example: 
<replace file="Deployment_ProjectAutomator.xml" token="%%OUTPUT%%" 
value="${build.output.dir}"/>
Above line signifies %%OUTPUT%% in the Deployment_ProjectAutomator.xml to be replaced with build output 
directory
${build.output.dir} -> will be passed as argument while executing ant script
${host} -> will be taken from /CICDDemo/deployer/config.cnf
${port} -> will be taken from /CICDDemo/deployer/config.cnf
${pwd} -> will be taken from /CICDDemo/deployer/config.cnf
${user} -> will be taken from /CICDDemo/deployer/config.cnf
${useSSL} -> will be taken from /CICDDemo/deployer/config.cnf
Command Syntax:
<SAG_Install_Directory>/common/lib/ant/bin/ant.bat -f  <updateAutomator.xml_File_location> -
Dbuild.output.dir=<Build_Output_Directory>


Example Command:
C:/Programs/SoftwareAG/common/lib/ant/bin/ant.bat -f C:\Users\WDAGUtilityAccount\workspaces\wm-
workspace\CICDDemo\deployer\updateAutomator.xml -
Dbuild.output.dir=C:\Users\WDAGUtilityAccount\workspaces\cicd-workspace\builds

In this exercise we will create a Jenkins job to automate all the steps from exercise 2-5
Commit Test Suite Executor to Git

Commit the Test Suite Executor project created in exercise 3 to Git repo along with the packages to deploy

Create Environmental variables in Jenkins
To create environmental variables: Click 'Manage Jenkins' > 'Configure System'
 
Under Global properties, enable Environment variables and add below variables
IS_HOST = Hostname of the IS with ABE installed
IS_PORT = Port of the IS with ABE installed
IS_USER_NAME = User to connect IS with ABE installed
IS_USER_PWD = Password to connect IS with ABE installed
SAG_INSTALL_DIR = Directory in with IS with ABE installed
WM_DEPLOYER_DIR = WmDeployer package directory
  
 
Configure JGit in Jenkins

Click 'Manage Jenkins' > 'Global Tool Configuration'
 
Under Git add JGit
 

Create freestyle Jenkins job

To create freestyle Jenkins job: New Item > Provide Job name (without space) > Freestyle project
 
Under General tab, you can use parameterized project to deploy only selected package
 
Connect to your Git repository under Source Code Management tab
 
Under Build tab, add a Execute windows batch command step and put the ABE build command from exercise 2
If you have the job parameterized to deploy only specific package, then use below
%SAG_INSTALL_DIR%/common/AssetBuildEnvironment/bin/build.bat -Dbuild.output.dir=%WORKSPACE%/builds -
Dbuild.source.project.dir=%WORKSPACE%/%PackageName% -Denable.build.IS=true
If you have the job non parameterized to deploy all packages in repo, then use below
%SAG_INSTALL_DIR%/common/AssetBuildEnvironment/bin/build.bat -Dbuild.output.dir=%WORKSPACE%/builds -
Dbuild.source.dir=%WORKSPACE% -Denable.build.IS=true
 
Under Build tab, add another Execute windows batch command step and put the command to update the 
ProjectAutomator.xml using ant script from exercise 3
%SAG_INSTALL_DIR%/common/lib/ant/bin/ant.bat -f %WORKSPACE%/CICDDemo/deployer/updateAutomator.xml -
Dbuild.output.dir=%WORKSPACE%/builds
 
Under Build tab, add another Execute windows batch command step and put the command to create Deployment 
project using projectautomator script from exercise 4
%WM_DEPLOYER_DIR%/bin/projectautomator.bat %WORKSPACE%/CICDDemo/deployer/Deployment_ProjectAuto
mator.xml
 
Under Build tab, add another Execute windows batch command step and put the command to deploy assets to 
target server using deployer script from exercise 5
%WM_DEPLOYER_DIR%/bin/Deployer.bat --deploy -dc Deployment_Candidate -project Deployment_Project -
host %IS_HOST% -port %IS_PORT% -user %IS_USER_NAME% -pwd %IS_USER_PWD%
 

Save the job and trigger the job by clicking 'Build now' (if not parameterized) or 'Build with Parameters' (if 
parameterized)
 
Monitor the job by clicking on the build number and then Console Output
 
After the job executed successfully, verify the package deployed in the IS with ABE installed (since the deployer 
server and the target server in the ProjectAutomator.xml is same).

Exercise 8: Create pipeline style Jenkins job to automate CICD process

In this exercise we will automate the build and deploy as soon as changes are pushed to repository using Jenkins job 
in pipeline style. Below flow diagram illustrates this exercise.
 
Jenkins pipeline script

A Jenkinsfile is a text file that stores the entire workflow as code and it can be checked into a SCM.
The Jenkinsfile is written using the Groovy DSL.
The Jenkins pipeline is written based on two syntaxes, namely:
1.	Declarative pipeline syntax
The declarative pipeline is defined within a block labelled 'pipeline' 
2.	Scripted pipeline syntax
The scripted pipeline is defined within a 'node'
Pipeline concepts
Pipeline
This is a user defined block which contains all the processes such as build, test, deploy, etc. It is a 
collection of all the stages in a Jenkinsfile. All the stages and steps are defined within this block. It 
is the key block for a declarative pipeline syntax.
 
Agent
An agent is a directive that can run multiple builds with only one instance of Jenkins. This feature 
helps to distribute the workload to different agents and execute several projects within a single 
Jenkins instance. It instructs Jenkins to allocate an executor for the builds.
A single agent can be specified for an entire pipeline or specific agents can be allotted to execute 
each stage within a pipeline. Few of the parameters used with agents are:
Any
Runs the pipeline/ stage on any available agent.
None
This parameter is applied at the root of the pipeline and it indicates that there is no global 
agent for the entire pipeline and each stage must specify its own agent.
Stages
This block contains all the work that needs to be carried out. There can be more than one stage 
within this directive. Each stage performs a specific task.
 
 Steps 
A series of steps can be defined within a stage block. These steps are carried out in sequence to 
execute a stage. There must be at least one step within a steps directive. In the following example 
I've implemented an echo command within the build stage. This command is executed as a part of 
the 'Build' stage.
