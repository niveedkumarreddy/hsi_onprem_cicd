# webMethods CI/CD Framework - Glossary

## Common Acronyms and Terms

### BDA - Build Deploy Automate
**BDA** is a naming convention used throughout the framework for property prefixes. It stands for **Build, Deploy, and Automate** - the three core functions of the CI/CD pipeline.

**Usage in Properties:**
```properties
bda.projectName=hsi_onprem_cicd      # Project identifier
bda.buildNumber=123                   # Build number from CI/CD
bda.targetEnv=DEV                     # Target environment (DEV/TEST/PROD)
```

**Why BDA?**
- Provides consistent naming across all configuration files
- Groups related properties together
- Makes it clear these are CI/CD automation properties
- Distinguishes from Software AG's native properties

## Core Framework Terms

### ABE - Asset Build Environment
Software AG's tool for building webMethods packages into deployable artifacts.

**Location:** `${SAG_HOME}/common/AssetBuildEnvironment`

**Purpose:**
- Compiles webMethods packages
- Creates File-Based Repositories (FBR)
- Validates package dependencies
- Generates deployment artifacts

**Example:**
```bash
# ABE is invoked by cicd/build-abe.xml
ant -f build.xml build -Dconfig.build.abeHome=${SAG_HOME}/common/AssetBuildEnvironment
```

---

### FBR - File-Based Repository
A directory structure containing built webMethods packages ready for deployment.

**Structure:**
```
tmp/fbr/hsi_onprem_cicd_123/
├── build.xml                    # Generated by ABE
├── MyBusinessPackage/
│   ├── manifest.v3
│   └── ns/
└── repository.xml
```

**Purpose:**
- Storage format for built packages
- Input for Deployer
- Portable deployment artifacts
- Version-specific builds

**Properties:**
```properties
fbrRepoName=hsi_onprem_cicd_123
fbrRepoDir=tmp/fbr/hsi_onprem_cicd_123
```

---

### PA - Project Automator
Command-line tool for automated deployment project creation and execution.

**Location:** `${SAG_HOME}/IntegrationServer/instances/default/packages/WmDeployer/bin/projectautomator.sh`

**Purpose:**
- Creates deployment projects programmatically
- Executes deployments via CLI
- Supports variable substitution
- Enables CI/CD automation

**Generated XML:**
```xml
<DeploymentProject>
  <ProjectName>DEV_123_hsi_onprem_cicd</ProjectName>
  <DeploymentSet>...</DeploymentSet>
</DeploymentProject>
```

**Execution:**
```bash
projectautomator.sh -f Deployment_ProjectAutomator.xml
```

---

### VarSub - Variable Substitution
Process of replacing placeholder values with environment-specific configurations during deployment.

**Files:** `varsub/DEV/`, `varsub/TEST/`, `varsub/PROD/`

**Example:**
```properties
# varsub/DEV/MyBusinessPackage.vs.properties
db.url=jdbc:sqlserver://dev-db:1433;databaseName=DevDB
api.endpoint=https://dev-api.company.com/v1
```

**In Package:**
```
Connection URL: ${db.url}
API Endpoint: ${api.endpoint}
```

**After Deployment:**
```
Connection URL: jdbc:sqlserver://dev-db:1433;databaseName=DevDB
API Endpoint: https://dev-api.company.com/v1
```

## webMethods Components

### IS - Integration Server
The core webMethods runtime environment that executes integration flows and services.

**Default Port:** 5555  
**Admin URL:** `http://server:5555`

**Components:**
- Packages (business logic)
- Services (integration flows)
- Adapters (connectivity)
- Triggers (event processing)

---

### UM - Universal Messaging
Software AG's messaging middleware for asynchronous communication.

**Default Port:** 9000  
**Purpose:**
- Message queuing
- Publish/Subscribe
- Event-driven architecture
- Guaranteed delivery

---

### WmDeployer - Deployer Package
webMethods package that provides deployment automation capabilities.

**Location:** `${SAG_HOME}/IntegrationServer/instances/default/packages/WmDeployer`

**Tools:**
- Project Automator (CLI)
- Deployer UI (web interface)
- Repository management
- Deployment history

---

### WmTestSuite - Testing Framework
webMethods package for automated testing of integration services.

**Test Types:**
- Unit tests
- Integration tests
- Mock services
- Assertions

**Output:** JUnit XML format for CI/CD integration

---

## CI/CD Terms

### Self-Hosted Runner
GitHub Actions runner installed on your own infrastructure (not GitHub's cloud).

**Why Self-Hosted?**
- Access to on-premise webMethods installation
- Network access to internal servers
- Control over build environment
- Security compliance

**Labels:** `[self-hosted, webmethods-build]`

---

### Environment Protection Rules
GitHub feature to control deployments to specific environments.

**Configuration:**
- **DEV:** No approval (auto-deploy)
- **TEST:** 1 approver required
- **PROD:** 2 approvers + 30-minute wait timer

---

### Matrix Strategy
GitHub Actions feature for parallel or sequential job execution.

**Example:**
```yaml
strategy:
  matrix:
    server: [test1, test2, test3]
  max-parallel: 1  # Sequential deployment
```

**Purpose:**
- Deploy to multiple servers
- Control deployment order
- Fail-fast or continue-on-error

---

### Artifact Retention
How long GitHub Actions stores build artifacts.

**Default:** 30 days  
**Contains:** FBR, build logs, test reports

**Purpose:**
- Rollback capability
- Audit trail
- Troubleshooting

---

## Configuration Properties

### Common Property Prefixes

| Prefix | Purpose | Example |
|--------|---------|---------|
| `bda.*` | Build/Deploy/Automate settings | `bda.projectName` |
| `config.build.*` | Build configuration | `config.build.abeHome` |
| `config.deployer.*` | Deployer settings | `config.deployer.deployerHost` |
| `config.test.*` | Test configuration | `config.test.reportDir` |
| `is*` | Integration Server paths | `isPackagesDir` |
| `fbr*` | File-Based Repository | `fbrRepoDir` |

## File Extensions

### .v3 - Manifest Version 3
webMethods package manifest file format.

**Location:** `packages/MyPackage/manifest.v3`

**Contains:**
- Package metadata
- Version information
- Dependencies
- Enabled/disabled status

---

### .ndf - Node Definition File
webMethods service or flow definition.

**Location:** `packages/MyPackage/ns/mycompany/services/node.ndf`

**Contains:**
- Service metadata
- Input/Output signatures
- Service type (flow, Java, etc.)

---

### .xml - Flow Implementation
webMethods flow service implementation.

**Location:** `packages/MyPackage/ns/mycompany/services/flow.xml`

**Contains:**
- Flow steps
- Transformations
- Branching logic
- Error handling

---

### .properties - Configuration Files
Key-value pairs for configuration.

**Types:**
- `build.properties` - Build configuration
- `*.vs.properties` - Variable substitution
- `deploy.properties` - Deployment settings

---

### .groovy - Groovy Scripts
Scripts written in Groovy language.

**Usage:**
- `ENV.groovy` - Environment definitions (DSL)
- `GenerateProjectAutomator.groovy` - PA XML generator

---

## Deployment Terms

### Deployment Candidate
A package or asset to be deployed to a target server.

**In Project Automator:**
```xml
<DeploymentCandidate>
  <SourceRepositoryName>hsi_onprem_cicd_123</SourceRepositoryName>
  <MapSetName>MyBusinessPackage</MapSetName>
  <TargetServerAlias>DEV_IS</TargetServerAlias>
</DeploymentCandidate>
```

---

### Deployment Set
A collection of deployment candidates deployed together.

**Purpose:**
- Group related packages
- Deploy as atomic unit
- Maintain consistency

---

### Target Server Alias
Logical name for a target Integration Server.

**Example:**
```groovy
DEV_IS {
    host = "dev-server.company.com"
    port = 5555
}
```

**Maps to:** Physical server in deployment

---

### Sequential Deployment
Deploying to servers one at a time (not parallel).

**Why?**
- Minimize risk
- Monitor each deployment
- Stop on first failure (PROD)
- Gradual rollout

**Implementation:**
```yaml
strategy:
  matrix:
    server: [prod1, prod2, prod3]
  max-parallel: 1
```

---

## Adapter Terms

### JDBC Adapter
webMethods adapter for database connectivity.

**Purpose:**
- Execute SQL queries
- Database transactions
- Connection pooling
- Multiple database support

**Configuration:**
```groovy
jdbcAdapters {
    DEV_JDBC {
        connectionAlias = "DevDB"
        driverClass = "com.microsoft.sqlserver.jdbc.SQLServerDriver"
        url = "jdbc:sqlserver://dev-db:1433;databaseName=DevDB"
    }
}
```

---

### AS400 Adapter
webMethods adapter for IBM AS/400 (iSeries) connectivity.

**Purpose:**
- Call AS/400 programs
- Access DB2/400 databases
- Data queue operations
- Message queue operations

**Configuration:**
```groovy
as400Adapters {
    DEV_AS400 {
        connectionAlias = "DevAS400"
        host = "dev-as400.company.com"
        port = 8471
    }
}
```

---

## ANT Terms

### Target
A named task or group of tasks in ANT build file.

**Example:**
```xml
<target name="build" depends="init">
    <!-- Build tasks -->
</target>
```

**Common Targets:**
- `build` - Build packages
- `deploy` - Deploy to environment
- `test` - Run tests
- `clean` - Clean temporary files

---

### Property
A variable in ANT build files.

**Definition:**
```xml
<property name="project.name" value="hsi_onprem_cicd"/>
```

**Usage:**
```xml
<echo message="Building ${project.name}"/>
```

**From File:**
```xml
<property file="build.properties"/>
```

---

## GitHub Actions Terms

### Workflow
A YAML file defining CI/CD pipeline.

**Location:** `.github/workflows/webmethods-cicd.yml`

**Contains:**
- Trigger events
- Jobs
- Steps
- Environment variables

---

### Job
A set of steps that execute on the same runner.

**Example:**
```yaml
jobs:
  build:
    runs-on: [self-hosted, webmethods-build]
    steps:
      - name: Checkout
        uses: actions/checkout@v4
```

---

### Step
A single task within a job.

**Types:**
- Action (uses: actions/checkout@v4)
- Command (run: ant build)

---

### Artifact
Files produced by a workflow that are stored for later use.

**Example:**
```yaml
- uses: actions/upload-artifact@v4
  with:
    name: build-artifacts-123
    path: tmp/fbr/**
```

---

### Environment
A deployment target with protection rules.

**Example:**
```yaml
environment:
  name: PROD
  url: https://prod1-server.company.com:5555
```

---

## Security Terms

### GitHub Secrets
Encrypted environment variables stored in GitHub.

**Access:** Settings → Secrets and variables → Actions

**Usage:**
```yaml
env:
  DEPLOYER_PASSWORD: ${{ secrets.DEPLOYER_PASSWORD }}
```

**Never:**
- Commit secrets to repository
- Log secrets in output
- Share secrets in plain text

---

### Branch Protection
Rules that control how code can be merged to protected branches.

**Common Rules:**
- Require pull request reviews
- Require status checks
- Restrict who can push
- Require signed commits

---

## Troubleshooting Terms

### Verbose Mode
Detailed logging for debugging.

**ANT:**
```bash
ant -f build.xml build -verbose
```

**Output:** Shows all property values, task execution, file operations

---

### Debug Mode
Even more detailed logging than verbose.

**ANT:**
```bash
ant -f build.xml build -debug
```

**Output:** Includes internal ANT operations, dependency resolution

## Quick Reference

### Most Important Terms

1. **BDA** - Build Deploy Automate (property prefix)
2. **ABE** - Asset Build Environment (builds packages)
3. **FBR** - File-Based Repository (build output)
4. **PA** - Project Automator (deployment CLI)
5. **VarSub** - Variable Substitution (environment config)
6. **IS** - Integration Server (runtime)
7. **WmDeployer** - Deployer package (deployment tool)

```
[prefix].[category].[specific]
   ↓        ↓          ↓
  bda   .targetEnv  =  DEV
config .deployer   .deployerHost = deployer.company.com
config .test       .reportDir = report/
```
